# Review Driven Coding

## 作用

这个 skill 用来在“编码阶段”提前落实复审标准，而不是等代码写完后再被动发现问题。

当用户要求你实现功能、修复 Bug、重构代码、补类型、补测试，并且期望结果能满足较高代码质量时，使用这个 skill。

## 目标

你生成的代码必须同时满足以下 5 个维度：

1. 稳定性
2. 性能
3. 设计合理性
4. 灵活可扩展性
5. 工程化质量

你的职责不是只让代码“可运行”，而是尽量让代码在交付前就达到 `review.md` 同等级别的要求。

## 工作流

### 1. 先理解，再动手

- 读取目标文件、直接依赖文件、类型定义文件、配置文件和同目录同类文件
- 明确改动范围、输入输出契约、已有模式、复用点与潜在风险
- 如果用户目标不明确，先澄清，不要带着模糊假设写代码

### 2. 编码前先做 5 维设计检查

#### 稳定性

- 考虑空值、undefined、边界值、异常数据、超时与失败路径
- 检查是否存在静默失败、未处理异常、竞态条件、资源泄漏
- 对输入做必要的防御式校验

#### 性能

- 避免热路径中的重复计算、重复创建对象、重复正则编译
- 频繁查找优先考虑 `Map` / `Set`
- 警惕无上限缓存、闭包持有大对象、逐条 I/O
- 避免不必要的同步阻塞和重复渲染

#### 设计合理性

- 保持职责单一、边界清晰
- 命名、参数、返回值要表达意图
- 避免魔法数字、魔法字符串和隐式副作用
- 保持抽象层次一致，不把高层业务和底层细节混在一起

#### 灵活可扩展

- 可能变化的值应抽为常量、参数或配置
- 扩展新能力时，应尽量通过扩展而非修改核心逻辑
- 类型定义要足够精确，并保留合理演进空间
- 避免过度设计，也避免把明显会变化的逻辑写死

#### 工程化质量

- 优先复用已有公共模块、类型和常量，避免重复逻辑
- 避免无理由使用 `any`、宽泛 `unknown`、粗暴类型断言
- 检查文件命名、导出方式、目录结构、构建配置是否符合项目约定
- 判断是否需要同步更新测试、配置、文档或类型文件

### 3. 实现代码

- 先说明计划修改哪些文件，再开始修改
- 优先做最小充分改动
- 只有在重复逻辑已经明确存在时，才提取公共抽象
- 如果发现现有架构本身有明显问题，要指出并给出兼容方案
- 默认遵循目标文件及同目录已有写法，减少因风格、结构、命名偏离导致的 lint / format 失败
- 未确认规则前，不主动做大规模重构、跨文件搬运、无关格式调整，避免把改动面扩大到校验链之外
- 对可能触发规则的点提前约束：控制函数长度与复杂度、避免无意义类型断言、避免 `== null`、避免未使用变量与导入、避免重复逻辑

### 4. 完成后强制自检

对所有新增或修改的文件执行自检：

- 稳定性：异常路径、边界、防御、资源释放
- 性能：热路径、重复计算、数据结构、阻塞点
- 设计：职责、命名、副作用、抽象层级
- 可扩展性：硬编码、扩展点、类型演进
- 工程化：DRY、类型精度、构建耦合、配置同步

发现真实问题时，先修复，再交付。

### 5. 验证

- 检查编辑文件的诊断信息
- 有条件时运行 lint、TypeScript 检查和最小相关测试
- 如果由于环境原因无法验证，要明确说明风险
- 优先对“本次修改涉及的文件”执行最小范围校验，而不是一开始跑全量校验
- 如果项目存在 pre-commit、lint-staged、格式化或类型检查链路，交付前要尽量按该链路验证一次
- 若校验失败，先根据报错做定点修复，不要在未确认根因前继续扩散改动
- 发现本地无法直接运行某个校验命令时，应改用项目脚本、包管理器脚本或等价入口完成验证

## 输出要求

最终答复应包含：

1. 修改了哪些文件
2. 为什么这样改
3. 如何满足 5 个维度
4. 做了哪些验证
5. 还有哪些风险或待补项

## 禁止事项

- 只实现 happy path
- 只追求功能跑通，不处理边界和失败路径
- 无理由引入重复代码
- 滥用 `any`、`as`
- 把配置、端点、阈值硬编码进核心逻辑
- 明知存在高置信问题仍直接交付
- 未经确认就顺手修改无关文件、无关格式或无关 import 排序，导致额外校验失败
- 在未执行最小必要校验前，直接声称“已完成”或“可提交”

## 示例触发

- “帮我实现这个页面，并确保代码质量高”
- “修这个 Bug，顺便把代码整理到能过 review 的水平”
- “重构这个模块，别只改表面”
- “给我生成一版符合 `review.md` 标准的实现”
